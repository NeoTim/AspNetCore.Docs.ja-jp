## <a name="introduction-to-integration-tests"></a><span data-ttu-id="fea06-116">統合テストの概要</span><span class="sxs-lookup"><span data-stu-id="fea06-116">Introduction to integration tests</span></span>

<span data-ttu-id="fea06-117">統合テストでは [単体テスト](/dotnet/core/testing/)よりも広範なレベルでアプリのコンポーネントを評価します。</span><span class="sxs-lookup"><span data-stu-id="fea06-117">Integration tests evaluate an app's components on a broader level than [unit tests](/dotnet/core/testing/).</span></span> <span data-ttu-id="fea06-118">単体テストは、個々のクラス メソッドなど、分離されたソフトウェア コンポーネントをテストするために使用されます。</span><span class="sxs-lookup"><span data-stu-id="fea06-118">Unit tests are used to test isolated software components, such as individual class methods.</span></span> <span data-ttu-id="fea06-119">統合テストでは、2 つ以上のアプリ コンポーネントが、連携して予想される結果を生成することを確認します。これは、要求を完全に処理するために必要なすべてのコンポーネントを含む可能性があります。</span><span class="sxs-lookup"><span data-stu-id="fea06-119">Integration tests confirm that two or more app components work together to produce an expected result, possibly including every component required to fully process a request.</span></span>

<span data-ttu-id="fea06-120">これらの広範なテストは、アプリのインフラストラクチャとフレームワーク全体をテストするために使用されます。多くの場合、次のコンポーネントが含まれます。</span><span class="sxs-lookup"><span data-stu-id="fea06-120">These broader tests are used to test the app's infrastructure and whole framework, often including the following components:</span></span>

* <span data-ttu-id="fea06-121">データベース</span><span class="sxs-lookup"><span data-stu-id="fea06-121">Database</span></span>
* <span data-ttu-id="fea06-122">ファイル システム</span><span class="sxs-lookup"><span data-stu-id="fea06-122">File system</span></span>
* <span data-ttu-id="fea06-123">ネットワーク アプライアンス</span><span class="sxs-lookup"><span data-stu-id="fea06-123">Network appliances</span></span>
* <span data-ttu-id="fea06-124">要求 - 応答パイプライン</span><span class="sxs-lookup"><span data-stu-id="fea06-124">Request-response pipeline</span></span>

<span data-ttu-id="fea06-125">単体テストでは、インフラストラクチャ コンポーネントの代わりに、*フェイク*または*モック オブジェクト*と呼ばれる、作成済みのコンポーネントを使用します。</span><span class="sxs-lookup"><span data-stu-id="fea06-125">Unit tests use fabricated components, known as *fakes* or *mock objects*, in place of infrastructure components.</span></span>

<span data-ttu-id="fea06-126">単体テストと比較すると、統合テストは次のようになります。</span><span class="sxs-lookup"><span data-stu-id="fea06-126">In contrast to unit tests, integration tests:</span></span>

* <span data-ttu-id="fea06-127">アプリが運用環境で使用する実際のコンポーネントを使用します。</span><span class="sxs-lookup"><span data-stu-id="fea06-127">Use the actual components that the app uses in production.</span></span>
* <span data-ttu-id="fea06-128">より多くのコードとデータ処理が必要です。</span><span class="sxs-lookup"><span data-stu-id="fea06-128">Require more code and data processing.</span></span>

<span data-ttu-id="fea06-130">そのため、統合テストの使用は最も重要なインフラストラクチャ シナリオに限定します。</span><span class="sxs-lookup"><span data-stu-id="fea06-130">Therefore, limit the use of integration tests to the most important infrastructure scenarios.</span></span> <span data-ttu-id="fea06-131">単体テストと統合テストのどちらを使用しても動作をテストできる場合は、単体テストを選択します。</span><span class="sxs-lookup"><span data-stu-id="fea06-131">If a behavior can be tested using either a unit test or an integration test, choose the unit test.</span></span>

> [!TIP]
> <span data-ttu-id="fea06-132">統合テストは、データベースとファイル システムを使用するデータおよびファイル アクセスのすべての順列として記述してはいけません。</span><span class="sxs-lookup"><span data-stu-id="fea06-132">Don't write integration tests for every possible permutation of data and file access with databases and file systems.</span></span> <span data-ttu-id="fea06-133">通常は、アプリ全体でデータベースやファイル システムを操作する場所がいくつあったとしても、的を絞った一連の読み取り、書き込み、更新、削除の統合テストを行うことで、データベースおよびファイル システム コンポーネントを適切にテストすることができます。</span><span class="sxs-lookup"><span data-stu-id="fea06-133">Regardless of how many places across an app interact with databases and file systems, a focused set of read, write, update, and delete integration tests are usually capable of adequately testing database and file system components.</span></span> <span data-ttu-id="fea06-134">これらのコンポーネントと連携するメソッドのロジックのルーチン テストには、単体テストを使用します。</span><span class="sxs-lookup"><span data-stu-id="fea06-134">Use unit tests for routine tests of method logic that interact with these components.</span></span> <span data-ttu-id="fea06-135">単体テストでは、インフラストラクチャのフェイク/モックを使用することにより、テストの実行時間が短縮されます。</span><span class="sxs-lookup"><span data-stu-id="fea06-135">In unit tests, the use of infrastructure fakes/mocks result in faster test execution.</span></span>

## <a name="aspnet-core-integration-tests"></a><span data-ttu-id="fea06-138">ASP.NET Core 統合テスト</span><span class="sxs-lookup"><span data-stu-id="fea06-138">ASP.NET Core integration tests</span></span>

<span data-ttu-id="fea06-139">ASP.NET Core の統合テストには、次のものが必要です。</span><span class="sxs-lookup"><span data-stu-id="fea06-139">Integration tests in ASP.NET Core require the following:</span></span>

<span data-ttu-id="fea06-144">統合テストでは、通常の *Arrange (配置)* 、*Act (実行)* 、および *Assert (確認)* のテスト ステップを含む一連のイベントに従います。</span><span class="sxs-lookup"><span data-stu-id="fea06-144">Integration tests follow a sequence of events that include the usual *Arrange*, *Act*, and *Assert* test steps:</span></span>

1. <span data-ttu-id="fea06-150">このプロセスは、すべてのテストが実行されるまで続行されます。</span><span class="sxs-lookup"><span data-stu-id="fea06-150">The process continues until all of the tests are executed.</span></span>
